# 企业年报工业互联网应用自动识别原型

一个基于 `Streamlit` 的本地 Web 工具，用于从企业年报 `.txt` 文件中自动提取与“企业是否真正应用了工业互联网平台”相关的候选句，并辅助完成 `0/1` 标注、人工审核与 Excel 导出。

这个项目最初面向一项课程作业场景：作业要求学生从企业年报中找出与工业互联网应用有关的句子，并判断这些句子是否表达了企业“已经在自身业务流程中真实落地应用”。手工逐篇查找和判断非常耗时，因此本工具尝试将这个流程做成一个可复用的半自动/自动化研究原型。

## 项目目标

- 批量读取企业年报 `.txt` 文件
- 尽量完整召回与工业互联网应用有关的候选句
- 用规则和大模型结合的方式判断句子标签
- 支持人工复核、修改标签和导出结果
- 尽量贴合作业要求，同时保留面向其他文本挖掘任务的可扩展性

## 核心思路

这个项目不是简单的“全文关键词搜索”，而是一个分阶段流水线：

1. 输入层
- 先指定一个年报总文件夹
- 再输入单个文件名或一组文件名
- 文件名按作业常见格式解析出 `证券代码 / 年份 / 企业名称 / 报告日期`

2. 读取层
- 自动尝试 `utf-16`、`utf-8-sig`、`utf-8`、`gb18030`、`gbk`
- 不再采用“谁先能解码就用谁”的粗暴策略
- 会对不同编码的解码结果做文本质量评分，尽量避免把 UTF-8 文本误判成 UTF-16 乱码

3. 预处理层
- 清除页码、目录、孤立噪声行
- 规范空白字符
- 按中文标点和换行切句
- 对“编号标题 + 正文”的写法做兼容
- 过滤明显只是标题而不是正文叙述的短句

4. 候选句召回层
- 一类关键词：作业原始词典
- 二类关键词：更宽松的兜底词典，用于提高召回率、避免某些年报完全没有候选句
- 关键词匹配前会做归一化，去空格、去常见标点，提高对断词和版式换行的容错能力

5. 规则判断层
- 识别未来规划词、自身应用动作词、对外服务词、效果词、平台建设词、工业场景词等
- 对明显的 `0` 和明显的 `1` 先做高置信度规则判断
- 其余候选句进入模型补判

6. 模型判断层
- 支持 `mock` 模式：不调用外部 API，只靠规则和启发式跑完整流程
- 支持 `DeepSeek API`：输出更接近人工批注风格的中文判断理由

7. 审核与导出层
- 支持在页面中逐条复核
- 可修改标签与备注
- 导出提交简表与分析详表

## 候选句如何抓取

候选句抓取不是直接在全文里生硬搜索关键词，而是按下面几步完成：

1. 先清洗文本
- 去掉页码、目录、释义等噪声行
- 统一空白字符和换行格式

2. 再切分句子
- 主要按 `。！？；` 和换行切句
- 对过长但没有明显句末标点的文本做补充切分

3. 处理标题干扰
- 过滤明显只是标题的短行
- 对“编号标题 + 正文”这种年报常见写法，尽量剥离标题，只保留正文参与匹配

4. 再做关键词匹配
- 先对句子做归一化，去空格、去常见标点
- 用一类关键词做基础召回
- 用二类关键词做兜底召回

5. 最后进入候选池
- 只有命中关键词的句子才进入候选池
- 候选句再结合规则特征和配额分配，进入后续预览与 AI 标注阶段

## 如何映射到作业要求

这套工具是按作业逻辑反向工程出来的，核心映射关系如下：

- 作业要求先找“命中关键词的句子”
  本项目通过一类关键词词典完成基础召回，再用二类关键词做兜底扩召回

- 作业要求判断企业是否“真正应用”
  本项目通过规则和大模型共同判断“自身应用 / 已落地 / 有动作 / 有效果”

- 作业要求排除规划、研发储备、对外服务、背景介绍
  本项目通过未来时态、对外服务、平台建设、背景介绍等规则标记降低优先级或直接判 `0`

- 作业要求每篇尽量有一定句子数，总量达到目标
  本项目支持设置“每篇期望句子数”“总最少句子数”“允许超额比例”，并对候选句做分配

## 特色功能

### 1. 两阶段工作流

不是一上来就调用模型，而是先做：

- `1. 先预览候选句`
- `2. 确认后开始 AI 标注`

这样可以先看：

- 一共抓到了多少句
- 分别来自哪些年报
- 每篇年报抓到了多少句
- 哪些年报偏少、偏多或存在编码风险

### 2. 编码鲁棒性

企业年报 `.txt` 来源复杂，编码经常不统一。这个项目专门做了多编码试读 + 文本质量评分，解决了“有些年报能读，有些年报一片乱码”的老问题。

### 3. 一类关键词 / 二类关键词双层召回

- 一类关键词：完全贴合作业原始要求
- 二类关键词：更贴近真实年报写法，负责兜底召回，防止某些年报只用一类关键词时检索不到句子

UI 中已经把这两种策略显式区分开：

- `一类关键词`
- `二类关键词兜底`

### 4. 标题与正文分离

很多年报写法是：

```text
3、人才联合培养机制成效显著
长春-苏州人才联合培养建立长效机制……
```

如果把这两行混在一起，容易影响切句与匹配。项目中专门加入了“标题前缀剥离”和“标题样式过滤”，让正文更稳定进入候选池。

### 5. 可解释判断理由

无论是规则还是 DeepSeek，最终都会给出中文理由，例如：

```text
这句话判定为1，原因是它明确描述了企业自身已经采取了落地动作，并且出现了可验证的工业场景或效果信号。
```

这样比只给一个 `0/1` 更方便人工审核。

### 6. 审核页区分一类 / 二类关键词命中

在审核页和分析详表中，命中关键词不再只混在一个字段里显示，而是拆分为：

- `一类关键词命中`
- `二类关键词命中`

这样可以更直观地区分：

- 句子是直接命中了作业原始关键词
- 还是依靠兜底关键词被召回进入候选池

这对人工审核尤其有帮助，因为可以快速判断某句话是“硬命中”还是“宽松召回”。

### 7. 面向人工复核的导出

导出分为两种：

- 提交简表
  包含 `句子内容 / 来源文件 / 人工标注标签 / id / year / 判断理由`

- 分析详表
  额外保留命中关键词、规则标记、上下文、置信度、模型来源、审核备注等信息

## 目前支持的判断模式

### `mock`

- 不调用外部模型
- 仅使用规则与启发式
- 适合联调流程、测试页面和导出功能

### `deepseek`

- 调用 DeepSeek Chat API
- 对边界句子的语义判断更自然
- 能返回更像人工标注员的判断理由

## 页面功能概览

### 1. 运行

- 设置年报总文件夹
- 输入单个文件名或批量文件名
- 选择判断模式
- 设置每篇期望句子数、总最少句子数、允许超额比例、关键词召回策略

### 2. 概览

- 查看文档总数、原始召回句数、最终候选句数、异常文件数
- 查看每篇年报的：
  - 原始召回数
  - 最终入选数
  - 目标句数
  - 编码
  - 解码告警
- 使用“年份 + 企业”方式展示可视化，便于阅读

### 3. 审核

- 按来源文件、标签、置信度筛选
- 查看句子上下文
- 区分查看一类关键词命中与二类关键词命中
- 修改人工复核标签与备注

### 4. 导出

- 下载双表 Excel
- 或直接写入本地导出路径

## 前端界面

当前前端基于 `Streamlit` 实现，采用“左侧参数配置 + 右侧分标签工作区”的布局：

- 左侧边栏负责输入路径、文件名、模型参数、配额参数和关键词召回策略
- 右侧主区域分为 `运行 / 概览 / 审核 / 导出` 四个标签页
- 概览页用于查看每篇年报的原始召回数、最终入选数、编码与状态
- 审核页用于逐条查看句子、判断理由、一类关键词命中和二类关键词命中

如果后续把界面截图文件放进仓库，例如 `docs/ui-overview.png`，可以继续把截图直接嵌进本 README。

## 安装与运行

### 环境要求

- Python 3.11 或 3.12
- Windows 本地环境优先测试

### 安装依赖

```bash
pip install -r requirements.txt
```

### 启动应用

```bash
streamlit run app.py
```

启动后在浏览器打开本地地址，例如：

```text
http://localhost:8501
```

## 使用示例

### 输入方式

先填写年报总文件夹，例如：

```text
F:\codex projects\Manual annotation of enterprise annual reports\enterprise annual reports
```

再在“年报文件名”中输入：

单文件模式：

```text
600148_2019_长春一东_2019年年度报告_2020-04-28.txt
```

批量模式，一行一个：

```text
600148_2019_长春一东_2019年年度报告_2020-04-28.txt
832982_2024_锦波生物_2024年年度报告_2025-04-21.txt
000001_2023_平安银行_2023年年度报告_2024-03-15.txt
```

### 推荐流程

1. 先选择 `二类关键词兜底`
2. 点 `1. 先预览候选句`
3. 在概览页检查每篇年报的原始召回数和最终入选数
4. 如果数量和分布都合适，再点 `2. 确认后开始 AI 标注`
5. 在审核页微调
6. 在导出页输出 Excel

  Excel 导出

## 测试

运行基础测试：

```bash
python -m unittest discover -s tests -v
```

当前测试覆盖了以下关键问题：

- UTF-16 读取
- 无 BOM 的 UTF-8 不被误判为 UTF-16
- 文件名解析
- 一类关键词匹配
- 二类关键词扩召回
- 一类 / 二类关键词命中字段输出
- 标题过滤

- 两阶段预览与标注流程

## 已知限制

- 目前输入文件限定为 `.txt`
- 标注质量仍受原始年报写法影响
- 二类关键词虽然能提高召回，但也会引入更多边界句
- DeepSeek API 需要用户自行提供 Key
- 当前是单机本地工具，不包含多人协作、数据库或在线部署能力

## 适用场景

- 企业年报中的工业互联网应用句子挖掘
- 面向课程作业的数据预标注
- 面向研究项目的候选句初筛
- 需要“规则 + 大模型 + 人工复核”结合的中文文本标注任务

## 后续可继续改进的方向

- 引入更细粒度的语义召回模型
- 支持 PDF 直接读取
- 支持跨年份同一企业相似句搜索
- 支持更多模型后端
- 支持批量审阅统计与标注质量评估

## 说明

本项目定位为研究与教学辅助工具。它可以帮助压缩手工检索和初筛成本，但最终结果是否用于正式作业提交，应由使用者自行确认是否符合课程要求与学术规范。
